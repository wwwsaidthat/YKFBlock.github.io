# 人机交互的历史

1. 命令行交互的方式
   1. 高效 可以自己定义命令
   2. 控制更加精准
   3. **节约系统资源**
   4. 要求用户对于键盘使用比较熟练，而且容易出错
   5. 基于记忆，对用户记忆的要求较高
   6. 使用贴近计算机的语言，违背不让用户了解计算机底层原理的可用性规则
   7. **可动态配置可操作选项**
2. 菜单
   1. 导航方式不够灵活
   2. 菜单规模较大，导航效率不高
   3. 占用空间
   4. 专家用户效率低 适合新手用户
3. 表格
   1. 占据空间，可能导致业务形式化
   2. 对于需要键入大量数据的比较合适
   3. 待填写区域已经预先定义了，只需要识别而无需记忆
   4. 简化输入
   5. 通过预定义格式来指导用户输入
4. 直接操纵的方式
   1. 基于识别 对初学者友好
   2. 有操作的快速反馈，用户的主观满意度较高
   3. 效率低
   4. 对屏幕显示性能要求较高，不适合小屏幕
   5. 实现较为复杂
   6. 自解释性差，容易误导用户
5. 隐喻
   1. 可扩展性差
   2. 设计师和用户的联想方式需要相似
   3. 有一些操作并没有合适的隐喻
   4. 把系统隐喻束缚于现实世界限制了软件的功能
6. 自然语言交互的方式
   1. 受限于自然语言的模糊性
   2. 计算机在处理理解语言所需要的上下文信息和先验知识方面还存在局限性
7. 批处理阶段 - 联机终端时代 - 图形用户界面GUI时期（直接操纵）

# 人机交互系统的原则和目标

1. 交互框架的作用

   1. 提供理解或定义某种事物的一种 结构 
   2. 能够帮助人们结构化设计过程 
   3. 认识设计过程中的主要问题 
   4. 还有助于定义问题所涉及的领域

2. EEC模型的指导意义

   1. 回答了软件为什么不好用的问题、
   2. 提出了执行隔阂和评估隔阂，并解决如何才能够使用户简单地确定哪些活动是被允许的、如何确定系统是否处于期望的运行状态等问题

3. 简化的可用性工程的四个技术

   1. 用户和任务观察

      1. 了解产品的目标用户是可用性工程的第一个步骤
      2. 注意
         1. 要直接与潜在用户进行接触 
         2. 不要满足于间接的接触和道听途说 
         3. “你”不是用户！

   2. 场景

      1. 简便易行的原型工具，通过省略整个系统的若干部分来减少实现的复杂性
      2. 水平原型：减少功能的深度并获得界面的表层 
      3. 垂直原型：减少功能的数量而对所选功能进行完整实现
      4. 这种工具格外节省时间和成本 场景的优点在于既能减少功能的深度又能减少功能的数量

   3. 边做边说

      是什么 让真实用户在使用系统执行一组特定任务的额时候，讲出他们的所思所想

      意义 最有价值的单个可用性方法

      作用 可以了解到用户为什么这样做，并确定其可能对系统产生的误解

      注意 实验的时候要不断的提醒用户，或者在实验之前看一段边做边说的视频让用户带入角色

   4. 询问专家之启发式评估--一种灵活而又相当廉价的评估方式

      优点

      1. 不涉及用户，所以面临的实际限制和道德问题很少
      2. 成本较低，不需要特殊设备 又被称为“经济评估法”

      缺点

      1. 评估人员需要长时间的训练才能成为专家
      2. 可能出现虚假警报，专家找到一个真实的可用性问题将发出约一个假警报忽略大约半个问题

      研究发现启发式评估可以发现很多可用性问题（剩下的可用性问题可以用边做边说方法来发现）

      为了避免个人的偏见应该让多个不同的人来进行经验评估

      注意事项 

      1. 邀请多个评估专家
         1. 评估专家不是用户 不同的评估专家可能发现不同的问题
         2. 越多越好，但是回报可能越来越小
         3. Nielsen推荐3-5名专家
      2. 使用用户测试替代启发式评估
         1. 不同方法发现的问题是不同的
         2. 虽然经济，但是用户测试才是黄金准则
      3. 观察人员可以帮助评估专家
         1. 专家用户可以但是用户测试就不适合了

      5名专家可以发现80%的问题，建议将测试分阶段进行

4. 启发式原则和其他原则的对应关系

   1. 尽可能保证一致  标准化和一致性
   2. 符合普遍可用性  使用的灵活性和高效性
   3. 提供信息丰富的反馈 系统状态的可见性
   4. 设计说明对话框以生成结束信息 系统状态的可见性
   5. 预防并处理错误  预防错误和帮助用户识别诊断和恢复错误
   6. 让操作容易撤销 自主权
   7. 支持内部控制点 控制权
   8. 减轻短时记忆负担 以来识别而非记忆

5. 启发式原则的应用

   1. 一致性和标准化 菜单文字和按钮提示的文字不一致
   2.  审美感和最小化设计 减轻用户的记忆负担
        很多可以点击的按钮看起来不像按钮
   3. 自主权和控制权 不支持撤销操作
   4. 审美感和最小化 比如冗余 谷歌的那个选择题 
      “主页上有太多选项”对应“审美与最小化设计”
   5. 一些操作规范和Windows操作规范不一致的地方 减轻用户负担
   6. 部分语言和用户常用术语不符 系统和现实世界的吻合

6. 如何正确评估

   1. 分析每个问题对应的启发式规则
   2. 列出所有问题
   3. 至少遍历两次界面
      1. 一次获得系统的初始体验 
      2. 另一次关注特定界面元素
   4. 不要局限于10条启发式规则

# 交互评估

1. 评估原则
   1. 评估应该依赖于产品的用户
   2. 评估与设计应结合进行
   3. 评估应该在用户实际工作任务和操作环境下进行
   4. 要选择有广泛代表性的用户
2. 评估范型
   1. 快速评估
      1. 设计人员非正式地向用户或顾问了解反馈信息，以正式设计构思是否符合用户的需要
         1. 可以在任何阶段进行
         2. 强调快速了解，而非仔细记录研究发现
         3. 得到的数据通常是非正式的、叙述性的
         4. 是设计网站时常用的方法
      2. 基本特征：快速
   2. 可用性测试
      1. 评测典型用户执行典型任务时的情况（包括用户出错的次数、完成任务的时间）
      2. 基本特征 是在评估人员的密切控制之下实行的
      3. 主要任务 量化表示用户的执行情况
      4. 缺点
         1. 测试用户数量通常比较少
         2. 不适合进行细致的统计分析
   3. 实地研究
      1. 基本特征：在自然工作环境中进行
      2. 目的：理解用户的实际工作情形以及技术对他们的影响
      3. 作用
         1. 探索新技术的应用契机
         2. 确定产品的需求
         3. 促进技术的引入
         4. 评估技术的应用
      4. 重难点
         1. 如何不对受试者造成影响
         2. 控制权在用户，很难预测即将发生和出现的情况
   4. 预测性评估
      1. 研究人员通过想象或者对界面使用过程进行建模
         1. 专家们根据自己对典型用户的了解预测可用性问题的可用性评估 
         2. 逐步通过场景或基于问题回答的走查法 
         3. 用于比较相同应用不同界面的原型法，如使用Fitts定律预测使用设备定位目标的时间 
      2. 基本特征
         1. 用户可以不在场
         2. 整个过程快速成本低
3. 区分评估技术的因素
   1. 评估在周期中的位置：设计在早期的评估更快速也更便宜
   2. 评估的形式：实验室环境or工作环境
   3. 技术的主客观程度：技术越主观，受评估人员知识的影响越大如认知走查
   4. 测量的类型 主观技术 定性数据 客观技术 定量数据
   5. 提供的信息 
      底层信息：这个图标是可理解的吗？ 
      高层信息：这个系统是可用的吗？
   6. 响应的及时性
      边做边说法可及时记录用户行为 
      任务后的走查取决于对事件的回忆
   7. 干扰程度
      直接观察可能会影响用户表现
   8. 所需资源
      设备、时间、资金、参与者、评估人员的专业技术及环境等
4. 启发式评估和用户测试的区别
   1. 前者不需要用户参与
   2. 两者发现的可用性问题不同可以互补

5. 实验设计题目看PPT

# 评估之观察用户

1. 观察适用于产品开发**的任何**阶段

2. 观察可以发现一些意想不到的用户的操作方式

3. 分成两种方式

   1. 真实环境中的观察，观察者可以作为旁观者也可以作为参与者，重点是应用的上下文
   2. 受控环境中的观察，参与者不能作为观察者，重点是研究用户执行任务的细节
   3. 二者的差别不大，有时候实地观察也可作为实验室观察的补充

4. 实验室观察

   1. 在专门为可用性测试而安装配置的固定设备的 环境下进行的观察 在一些情况下，在实验室中观察是唯一的选择 如空间站等危险环境下的系统

   2. 实验室布局 

      1. 测试区 

      2. 观察区 

         为避免干扰，二者分开 

   3. 用户坐在家中测试 

      1. 如Windows Live Meeting等软件 可观察到鼠标移动及屏幕内容的变化 
      2. 使用户处于更加自然和真实的状态

   4. 优点 提供了可控且一致的评估环境 易于分析比较 

   5. 缺点 

      1. 可控且一致的评估环境 
         1. 人为环境、不自然 
         2. 可能降低测试结论的普遍性和一般性 
      2. 不利于观察多人之间的协作 

      

# 评估之询问用户和专家

1. 访谈步骤 开始 热身 正式访谈阶段 冷却 结束
2. 访谈类型 
   1. 半结构化 结构化 非结构化
   2. 集体访谈 -- 焦点小组
      1. 非正式的评估方法
         1. 在界面设计之前和经过一段使用之后评估用户的需要和感受 
         2. 是市场、政治和社会科学研究经常使用的方法 
         3. 人数限制：由大约6到9个典型用户组成 
         4. 如在评估大学的网站时，可考虑由行政人员、教师和学生们组成3个分别的 焦点小组 
      2. 主持人工作 
         1. 事先列出一张讨论问题和数据收集目标的清单 
         2. 保持所谈论的内容不离题 
         3. 保证小组的每个成员都积极参与谈论 
         4. 讨论结果的分析报告 
      3. 焦点小组存在风险
3. 询问用户之问卷调查
4. 询问专家之认知走查
   1. 专家分析可应用于项目设计的**任何阶段**
   2. 逐步检查使用系统执行任务的过程，从中找出可用性问题 
      1. 无需用户参与 
      2. 认知走查的主要目标是确定使一个系统如何易于学习 
      3. 试图想象出人们在**第一次**使用某个产品时的想法以及所采取的动作，它的大致流程是怎样的
      4. 评估的具体过程就是把用户在完成这个功能时所做的所有动作讲述成一个令人可以信服的故事
   3. 走查的步骤
      1. 标识并记录典型用户的特性 有关用户自身心理、心理特点以及他们的知识和经验的描述 
      2. 基于评估重点，设计样本任务 应该是大多数用户要做的典型任务 
      3. 制作界面原型（或界面描述），明确用户执行任务的具体步骤 
      4. 由设计人员和专家级评估人员（一位或多位）共同进行分析 
      5. 评估人员结合应用的上下文，逐步检查每项任务的操作步骤  
      6. 在完成逐步检查之后，汇总关键信息 
      7. 修改设计，更正发现的问题
5. 协作走查
   1. 由用户、开发人员和可用性专家合作，逐步检查任务场景，讨论 与对话元素相关的可用性问题 
      1. 在评估过程中，每一位专家都承担用户的角色 
      2. 优点 
         1. 专注于用户任务；能够产生定量数据 
         2. 符合参与式设计原则 
      3. 缺点 
         1. 需要各方面的专家，速度慢 
         2. 由于时间限制，通常只能评估有限的场景
6. 询问专家之启发式评估见上面

# 评估之用户测试--DECIDE评估框架

1. 六个步骤 

   1.  决定评估需要完成的总体目标 

   2.  发掘需要回答的具体问题 

   3. 选择用于回答具体问题的评估范型和技术 

   4.  标识必须解决的实际问题，如测试用户的选择 

   5. 决定如何处理有关道德的问题 

   6. 评估解释并表示数据

2. 小规模试验

   1. 对评估计划进行小范围测试 
      1. 以确保评估计划的可行性 
      2. 如检查设备及使用说明 
      3. 练习访谈技巧 
      4. 检查问卷中的问题是否明确 
   2. 小规模试验可进行多次 
      1. 类似迭代设计 
      2. 测试——反馈——修改——再测试 
      3. 快速、成本低

3. 用户测试须考虑实际限制并做出适当的折衷 

   1. 应确保不同参与者的测试条件相同 
   2. 应确保评估目标特征具有代表性 
   3. 实验可重复，但通常不能得到完全相同的结果 
   4. 以DECIDE框架为基础

4. 测试设计步骤

   1. 定义目标和问题

   2. 选择参与者

   3. 设计测试任务

      1. 测试任务应当与定义的目标相关 

      2. 测试任务通常是简单任务 

         1. 如查找信息 

      3. 有时采用较为复杂的任务 

         1. 如加入在线社团等 

      4. 任务不能仅限于所要测试的功能，应使用户全面的使用设计的各个区域 

         1. 如关注搜索功能的可用性，可请求参与者搜索找出产品X 
         2. 更好的方法就是请求参与者找出产品X并同产品Y进行比较 

      5. 每项任务的时间应介于5~20分钟 

      6. 应当以某些合乎逻辑的方法安排任务 

         开始时，先提出简单问题有助于增强用户的自信心

   4. 明确测试步骤

      1. 在测试之前，准备好测试进度表和说明，设置好各种设备 
      2. 正式测试前应进行小规模测试 
      3. 在必要时，评估人员应询问参与者遇到了什么问题 
      4. 若用户确实无法完成某些任务，应让他们继续下一项任务 
      5. 测试过程应控制在1小时之内 
      6. 必须分析所有搜集到的数据

   5. 数据搜集

      1. 确定如何度量观测的结果 

      2. 使用的度量类型（定性/定量）依赖于所选择的任务

         **常用的定量度量** 

         完成任务的时间 

         停止使用产品一段时间后，完成任务的时间 

         执行每项任务时的出错次数和错误类型 

         单位时间内的出错次数 

         求助在线帮助或手册的次数 

         用户犯某个特定错误的次数 

         成功完成任务的用户数

5. 实例

   网站评估实例 

   在对MEDLINEplus网站进行启发式评价后 

   发现了可用性问题，对网站做了修改 

   现计划对网站进行用户测试 

   1：定义目标和问题 

   ​	信息分类方法是否有效 

   ​	用户能否进退自如并且找到需要的信息 

   2：选择参与者 

   ​	通过问卷了解年龄、使用互联网的经验、查找医药信息的频度 

   ​		挑选每个月使用互联网超过两次的人员 

   ​	9位来自测试中心所在地的医护人员/ 7名是女性 

   ​		符合可用性专家所建议的5-12位 

   ​	预先声明要测试NLM的一个产品

   3：设计测试任务 

   问题选自网站用户最经常提出的一些问题 

   设计了5项任务 

   ​	任务1：查找信息，了解肩膀上的黑痣有没有可能是皮肤癌 

   ​	任务3：查找信息，了解是否有丙肝疫苗 

   ​	进行了小规模试验以确定任务的有效性 

   4：明确测试步骤 准备统一的说明稿，分为五个部分 

   ​	以保证每一位参与者都得到相同的信息和相同的对待 

   ​	测试在实验室环境中进行

   5：数据搜集 

   ​	评估小组事先设定了成功完成每 项任务的标准 

   ​	如必须找到并访问3-9个相关网页 

   ​	记录用户执行任务的全过程 

   ​	以下为参与者A在执行第一项任务时访问的资源

   6：数据分析 

   ​	网站的结构，如专栏的安排、菜单的深度和链接的组织等 

   ​	浏览的有效性， 如菜单的使用、文字密度等。 

   ​	搜索特征，如搜索界面、提示、 术语的使用是否满足一致性要求

   7：总结、报告测试结果

# 基础知识

1. 人类处理机模型

   三个交互式组件

   感知处理器 信息将被输出到声音存储和视觉 存储区域 

   认知处理器 输入将被输出到工作记忆 

   动作处理器 执行动作

2. 格式塔心理学 五个原则和对应的启示

   1. 相近性原则 
      1. 合理运用接近性法则，它能让界面层次清晰有序
      2. 例如列表页设计，将相关的信息组合在一起并重复排列出来，就能明显感知 不同小组之间的界限，当同一小组内元素关系明确时，将其更加靠拢，用户 视觉就会更聚焦。
   2. 相似性原则
      1. 应用 
         使用不同的大小、颜色、形状来创建对比或视觉权重，呈现出不一样的视觉 效果，以达到弱化（降低视觉）或凸显（强化视觉）某些内容
   3. 连续性原则
      1. 共线或具有相同方向的物体会被组合在一起
      2. 将组件对齐，更有助于增强用户的主观感知效果
   4. 对称性原则
      1. 相互对称且能够组合为有意义单元的物体会被组合在一起
   5. 完整和闭合性原则
      1. 人们倾向于忽视轮廓的间隙而将其视作一个完整的整体 
      2. 页面上的空白可帮助实现分组

3. 感觉记忆 

   1. 又称瞬时记忆，在人脑中持续约为1秒钟 ，帮助我们把相继出现的一组图片组合成一个连续的图像序列，产生动态的影像信息

4. 短时记忆 

   1. 感觉记忆经编码后形成，又称工作记忆，约保持30秒 
   2. 储存的是当前正在使用的信息，是信息加工系统的核心，可理解为计算 机的内存 
   3. 短时记忆的存储能力约为7±2个信息单元

5. 长时记忆

   1. 短时记忆中的信息经进一步加工后会变为长时记忆 
   2. 只有与长时记忆区的信息具有某种联系的新信息才能够进入长时记忆，长时记忆的信息容量几乎是无限的 

6. 启发 

   1. 注意使用线索来引导用户完成特定任务 
   2. 在追求独特的创新设计时也应注重结合优秀的交互范型

# 以用户为中心

上下文询问的四个原则

1. 上下文环境

   应深入工作空间，以了解其中发生的事情 

   可以要求用户边做边说，也可以只在必要时发问

2. 伙伴关系

   开发人员和用户应相互合作 提醒用户是专家，将研究人员作为新手

3. 解释 解释过程必须由用户和开发人员合作完成 杜绝设计人员片面地对事实作出解释或假设

4. 焦点 把问题集中在所定的研究题目上，准备一个观察方向的列表

上下文询问法和观察法的区别

上下文询问法：边做边聊 

观察法：只看不说

以用户为中心的设计思想的局限性

1. 影响产品的创新性 
2. 可操作性受到时间、预算和任务规模的限制 
3. 忽视了人的主观能动性和对技术的适应能力
