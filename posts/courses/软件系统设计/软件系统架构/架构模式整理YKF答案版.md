1. 主机终端阶段
	1. 迁移原因：越来越多的人需要共享同一套核心数据和事务能力，批处理没办法满足
	2. 做法：通过“强中心+弱终端”换取一致性、安全性与治理能力，把事务正确和集中治理做到最强
	3. 取舍：主要解决很多人如何共享核心事务系统，但是交互的灵活性很弱
2. C/S阶段
	1. 迁移原因：用户要求更丰富、更快速的桌面交互
	2. 做法：把交互移动到客户端，把数据留在服务器
	3. 取舍：解决了怎样做富交互，但是把部署和版本复杂度也搬到了客户端
3. 三层/分层
	1. 迁移原因：降低部署复杂度，并把系统内部职责整理清楚
	2. 做法：中心从“前后分工”转向“服务端内部清晰分层”
	3. 取舍：解决了系统内部如何更清楚，但是并没有解决系统之间如何协同
	4. Springboot
4. SOA（Service-Oriented Architecture)
	1. 迁移原因：企业需要跨系统复用业务能力，并把端到端流程打通
	2. 做法：通过契约总线编排治理把企业从孤岛系统集合拉向了能力网络
	3. 取舍：解决全企业协同，代价是治理体系更重、中心中间件更强
	4. EBS：企业级总线
5. 微服务
	1. 迁移原因：团队要快速迭代，但是团队的重治理和粗粒度服务拖慢了速度
	2. 做法：通过更小的业务边界与独立部署，把交付链条明显缩短
	3. 取舍：微服务用边界更小换交付更快，把系统边界、团队边界和发布边界尽量对齐
	   复杂度并没有消失只是从代码内部转向了分布式协作、数据一致性与运维层
	4. 工具：DevOps CICD RESTful API
6. 云原生
	1. 迁移原因：同步链路过长、扩容缩容困难，人工运维不可持续,运行能力不平台化，没法形成规模效应
	2. 做法：异步事件和平台化运行，让系统更加弹性、更加韧性、更自动化
	3. 取舍：让系统在大规模分布式系统中更稳，但是也让时序推理、排障和最终一致性治理更难
	4. 工具：K8s Docker
