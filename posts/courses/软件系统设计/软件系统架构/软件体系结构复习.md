# 001 Introduction 
1. 架构和设计的异同是什么？
	1. 相同点：都是关于软件设计，软件设计包含架构设计和详细设计
	2. 不同点：
		1. 不是所有的决策都可以被称为是架构级别的设计，只有影响系统结构，影响质量属性，改动成本极高的决策才是架构决策，举个例子选springboot还是其他架构是架构决策，但是用ifelse还是某些设计模式这些是详细设计
		2. 从另外一个角度来看，架构设计是软件设计中的高位设计，但是不全是高位设计因为有的时候架构设计的粒度也会涉及到小的细节层面上面
		3. 架构是一系列设计决策的组合，架构包含元素 关系 和属性（元素 元素组 整个系统三个粒度）
2. 架构和结构有什么不同的？
	1. 结构只是分解，把大系统切分成小的零件
	2. 架构则不光是分解还定义了
		1. 模块的功能
		2. 模块间的通信和依赖
		3. 模块的职责
3. 组件间通信包含几个维度？
	包含两个正交维度
	1. 数据传递机制：组件A要把数据/请求交给组件B，通过什么方式
		1. 函数调用 function call（在工位上喊同事）
		2. 跨网络调用远程方法 Remote method invocation（打电话） 
		3. 消息队列/事件总线 Asynchronous message 不等结果（发邮件）
	2. 控制流：多个组件之间的消息流动，在时间轴上是怎么排列的
		1. Sequential 顺序执行 -- 慢
		2. Concurrent / parallel 多个组件同时执行
		3. Synchronization 并发时如何协调--如果并发的两个进程要同时修改一个文件怎么解决
4. 非功能性需求NFR包括什么？
	1. 技术限制
	2. 业务限制
	3. 质量属性
5. 为什么设计需要抽象？
	1. 隐藏底层的具体实现以及复杂性，只关注各个模块功能以及各个模块之间的交互
	2. 架构元素和软件元素不一定是一一对应的，因此需要进行抽象，方便做决策
6. 什么是黑盒设计？什么是白盒设计？两者之间有什么区别？
	1. 黑盒设计是只看接口，关心输入和输出，并不关心内部具体是怎么实现的
	2. 白盒设计则是可以看到具体的内部实现是怎么样的,进行设计时会参考具体的实现
7. 为什么需要多个架构的视图？
	1. 类似于建楼，你需要平面图，外立面图，电气图，给排水图
	2. 给不同的相关利益者看对应的视图

9. 4+1视图分别指什么，有什么用，一般的形式是什么样的，给谁看的？
	1. Logical View逻辑视图
		1. 关键功能组件、他们的职责、他们怎么交互
		2. UML类图
		3. 对象是客户、产品经理、业务分析师
	2. Process View进程视图
		1. 多线程/多进程、进程间通信、同步互斥死锁、消息传递、事件流
		2. 时序图、活动图
		3. 性能工程师、集成工程师
	3. Physical View 物理视图
		1. 软件组件和服务器节点容器之间的对应关系：
			1. 哪个服务跑在哪个机器上
			2. 数据库怎么分布，跨机房还是单机房
		2. 部署图
		3. 运维、SRE、系统管理员
	4. Development View开发视图
		1. 模块 子系统的划分以及它们之间的依赖关系
		2. 对象是开发人员
	5. Scenarios场景（视图）
		1. 走通四个视图
		2. 对象是所有人--验证架构一致性的工具
