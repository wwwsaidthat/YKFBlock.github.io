计划设计死于蓝图幻象，Code-fix设计死于纪律缺席。

计划设计

  • · 两群人说不同语言

纯 Code-fix 演进

设计随实现自然生长，但无约束。

  • · 持续埋雷
  • · 无测试，不敢重构
  • · 半年后无人敢于修改
  • · Kent Beck 说：狗屋能这么拼，摩天大楼不行
失败不在做不做设计，而在放任蓝图脱离代码或代码脱离设计。

敏捷测试的四条纪律

01
测试是设计工具，在判断力失效时给出判据。
1
独立 · Isolated 互不影响，可打乱顺序。
2
自动 · Automated 自动运行并判定，不依赖人工。
3
即时 · Immediate 编码前写，写测试即设计接口。
4
永存 · Persistent 永久保留、反复运行；套件即活文档。
判断力失效时，自动测试给出通过/不通过，这是 XP 最强的安全网。

红 · 绿 · 重构：TDD 飞轮

03
TDD 颠倒编码顺序：先定义完成，再让代码追上。

每轮循环仅做一件事，3 分钟可完成。

  • ①
    红 · 写失败测试
    例：assert add(2,3) == 5。实现未写，测试先红。
  • ②
    绿 · 最小代码过测
    return 5 即可，允许硬编码，目标仅让红变绿。
  • ③
    三角测量 · 再补一条测试
    ② 允许了 return 5 硬编码。加 add(3,4)==7，硬编码立刻红，被迫泛化成 a+b。
  • ④
    重构 · 在测试保护下
    重命名、抽函数、加类型，测试是安全绳。
PART 2 · 尝试 03
Part 2 · 尝试

add(2, 3) 的四步演进

04
用最小示例走完一轮红绿重构。
# 第 ① 步 — 写失败测试（红）
def test_add_two_and_three():
    assert add(2, 3) == 5 # NameError: 'add' is not defined
# 第 ② 步 — 最小代码让它变绿（允许硬编码）
def add(a, b):
    return 5 # 够让测试过就行
# 第 ③ 步 — 再补一条测试，逼出硬编码
def test_add_three_and_four():
    assert add(3, 4) == 7 # FAILS：硬编码被打脸
def add(a, b):
    return a + b # 被逼出真正的实现
# 第 ④ 步 — 在测试保护下重构（加类型、加校验）
def add(a: int, b: int) -> int:
    return a + b # 重命名、抽函数、加断言都安全
绿步允许不优雅，仅为让红灯灭掉。下一条测试会逼出丑陋。重构只在全绿时做。

05

简单设计的四重准则

06
四准则按优先级叠加，交集即简单设计。
  • ①
    通过所有测试
    功能真的管用。
  • ②
    消除重复
    相同概念只表达一次。
  • ③
    清晰表达意图
    名字和结构让下一个读者秒懂。
  • ④
    最小化元素数量
    前三条满足后能删就删。
① > ② > ③ > ④。冲突时先消除重复，重复是熵增温床。

YAGNI：You Aren't Gonna Need It

07
"以后可能要"是软件里最贵的四个字。

数据 1 · 使用率

< 20%

超前开发功能的真实使用率 < 20%。

剩余 80% 是维护负担。

数据 2 · 养护成本

≈ $4

每行预留代码年均维护成本。

1 万行预留 = $40k/年。

提前 N 周期开发，总成本 = C × (1+维护率)^N；实际需求到来时成本仅 C × 0.7。案例：某支付系统提前做加密货币接口，三年后需重构，总成本增加 230%。

重构的纪律：不改行为，只改结构

08
重构与"临时修改"的区别：有测试守护。
  • 🧼
    提升可读性
    重命名、拆分函数、合并重复。
  • 🪛
    降低维护成本
    消除代码异味。
  • 🌱
    支持演化设计
    小步响应变化，避免大爆炸。
  • 🔒
    测试保护是前提
    无测试先补测试，否则黑夜拆引擎。

配套工具链

  • ⚙️
    IDE 重构
    IntelliJ / Eclipse / VS Code：提取方法、内联变量、重命名。
  • 🧪
    测试框架
    JUnit / pytest / Jest：守住外部行为。
  • 🔍
    质量扫描
    SonarQube：检测重复代码、圈复杂度。
重构是保留行为的结构优化。改变可见行为的是功能变更，不是重构。

生长式架构：疼痛出现前不引入核心元素

01
架构是难以修改的核心元素。延迟决定哪些是核心，直到证据足够。
0
起步 · 什么都不上 不设数据库、领域模型、缓存；内存结构跑最小闭环。
1
海量数据到来 → 引入数据库 撑不住时才引入，不为"以后可能需要"。
2
业务复杂度到来 → 引入领域模型 if-else 嵌套到第三层，领域模型登场。
3
存疑时 · 依 YAGNI 没想清楚不做；架构拆出贵十倍。
Kent Beck · Ron Jeffries · Bob Martin 三人一致：避免前期架构，需要时再引入。Fowler 补充：海量数据用数据库，复杂业务用领域模型，其余 YAGNI。

UML 的两种用法：草图 vs 终稿

02
"有用则用"是 Fowler 的标准：沟通为主，代码才是全面信息源。

✅ 草图用法（提倡）

  • · 编码前用图表探索设计
  • · 保持简短，一两张白板就够
  • · 不解决所有细节，只画关键结构
  • · 视设计为草图而非终稿
  • · 讨论完即擦掉

❌ 终稿用法（抛弃）

  • · 作为蓝图交给另一团队构建
  • · 需要长期与代码同步维护
  • · 试图解决全部细节UML 做不到
  • · 代码改图也改，双倍工作量
  • · 图比代码权威，认知倒置
若一张图三周后无人看，便是浪费。白板拍照存档，不进 Git。

可逆性：把复杂性挤出系统

03
不可逆决策是复杂性的主要来源；降低不可逆性是敏捷的策略。
  • ①
    如果决策可更改，就不必一次做对
    把"猜"的成本压到近零，错了就换。版本控制的哲学根据。
  • ②
    推迟决策到信息最充分的时刻
    数据库选型、微服务拆分、状态管理，凡选后难改的都先拖。
  • ③
    让决策可逆转
    接口抽象、feature flag、适配层，每增加一个就多留一扇门。
  • ④
    版本控制是代码级的可逆保险
    敢 refactor 因为敢 revert；Git 是 XP 纪律的前提。
问：三个月后后悔，多快能撤回？撤不回的决定再想想。

Part 3 · 解法

设计未死——只是换了活法

05
设计未死，只是本质已变。XP 让演进式设计重获新生。
  • ①
    保持代码清晰简洁
    可读性是后续一切行为的前提。
  • ②
    通过重构改进代码
    小步、持续、测试保护下。设计分散到每一天。
  • ③
    精通设计模式：懂应用时机与演进路径
    模式是识别代码何时该长成什么样的词汇表。
  • ④
    预见变化
    今日决策终将改变，须留接口、feature flag、版本控制。
  • ⑤
    以代码、图表、对话传达设计
    三者皆沟通，代码是唯一权威；图表是草图，对话是共识。
Martin Fowler：优秀设计向来艰难。XP 把设计从一次性前期活动变为每天持续的活动。
