一个仓库里的"不知道还要多久"

01
Fowler 实习时见到的那一幕，是许多项目的真实写照。
"一家大型英国电子公司，我的经理来自质量保证团队。他带我走进一个巨大、沉闷、没有窗户的仓库，里面全是在小隔间里写代码的人。项目已经做了好几年，编码部分其实早就结束了——他们现在只是在'整合'，而且已经整合了好几个月。没人知道还要多久。" — Martin Fowler, Continuous Integration
这不是孤例。当集成被推迟到项目末尾，它就不再是一次活动，而是一段没人能估出工期的黑洞——每个模块单独跑都没问题，合在一起却总是不通。

传统集成会同时收三种税

02
不是"辛苦一阵子就过去了"，而是同时发生、互相放大的系统性损耗。

难定位的错误

  • 改动跨了几周，合并后出 bug
  • VCS 能报源代码冲突，报不了语义冲突
  • 排查范围从一次提交扩成几十次

上下文切换

  • 本该写功能，被迫调同事的冲突
  • 心流被打断，效率直接掉
  • 集成完成后回到本地，自己负责的部分也已遗忘大半

回避重构

  • 知道结构烂，不敢动
  • 一旦修改就引发连环冲突
  • 技术债利滚利，代码库带病运行

Part 2 · Practice

开发者的一日循环

01
每天至少走一圈的四步脚本——四步中有三步是"执行构建"。
1
从主线拉取最新代码 git pull origin main——让本地不落后，减少后续冲突的发生面。
2
本地修改 + 完整自动化构建 构建通过后再继续修改，修改完成后重新构建。构建包含编译、单元测试、静态检查——任何一项红就不推送。
3
推送前再 pull 一次、再构建一次 此时同事很可能已合并入新代码。合并后再执行完整构建，验证本地改动能否与同事的改动共存。
4
git push 推送至主线 触发中央 CI 在干净环境中再次运行——消除"在我机器上能跑"。
为什么重复执行三次构建？每一步失败都给出一个更小的怀疑半径——CI 的调试效率来自这个"小半径"，而非来自构建本身多聪明。

Part 2 · Practice

十条核心实践

02
Fowler 把这些实践分成显性的"动作"和隐含的"基础设施"——两组都齐，CI 才成立。
显性 · 动作（开发者日常要做） 隐含 · 基础设施（平台层保障）
① 维护单一主线代码库 (Mainline) ⑦ 在类生产环境中运行测试
② 自动化构建：一键完成编译打包 ⑧ 任何人都能方便取到最新可执行文件
③ 让构建自测试（构建内含测试） ⑨ 每个人都能看到当前构建状态
④ 每人每天向主线提交 ⑩ 自动化部署
⑤ 每次主线提交都触发构建
⑥ 保持构建快速
只做一组等于没做：只做动作会把团队累死；只搭基础设施是空架子。两组必须一起上。
PART 2 · PRACTICE 08
Part 2 · Practice

"能编译"≠"做对了事"

03
编译器的盲区很宽——逻辑错、边界漏、回归缺陷都溜得过去。

❌ 只靠编译通过

  • 静态类型抓类型错，抓不到逻辑错
  • 边界条件、并发顺序、异常分支多数漏过
  • 能 build ≠ 能跑对
  • 集成冲突在运行时才暴露

✅ 自测试构建

  • 构建 = 编译 + 完整自动化测试
  • 任一测试失败，整个构建变红
  • 可加 Linter、静态分析、覆盖率
  • 代码库能自己验证自己是否正确
Kent Beck 的朴素检验：想象有个"顽皮小鬼"在代码里注释掉一行、反转一个条件——健全的测试套件应当至少让一个测试失败。抓不住，就说明测试还不够。

不同生态，同一契约：一条命令完成完整构建

04
让任何人检出代码后，一条命令就能把系统装起来、测一遍。
# C / Unix $ make # Java · Maven $ mvn install # Java · Gradle $ ./gradlew build # Node.js $ npm run build && npm test # Python $ pip install -e . && pytest # .NET $ dotnet build && dotnet test
契约一致，工具无关：一条命令，一个结果——绿或红。让开发者手点 IDE 按钮、记忆一串奇怪参数，就是在用人肉对抗熵。

十分钟，是注意力的硬约束

06
提交构建超时，CI 的活力就断了——这并非规范问题，而是心理学问题。
"10 分钟"不是精确数字，而是一个心理阈值：一旦超过这条线，开发者就会放下手中的工作，转去刷手机、参加会议或处理其他事务。等构建失败的通知返回时，上下文早已丢失，修复成本数倍增加。
PART 2 · PRACTICE 12
Part 2 · Practice

构建坏了怎么办

07
Kent Beck 的规矩："主线坏了，谁也别想回家"——修主线是当前最高优先级。
1
首选：Revert 那次提交 git revert 让主线立刻回到上一个绿色状态——解除对所有其他人的阻塞。Revert 不是认输，是把故障半径压到最小。
2
在 revert 后的主线上重修 问题开发者在本地修改，充分测试后再重新集成。不要在已损坏的主线上继续提交新代码——这只会让问题相互纠缠。
3
特殊情况才"主线快修" 只有当问题简单到目测可控时才允许。另一种备选是"门禁提交 / Gated Check-in"：提交先进入临时区验证再合并，重但稳，团队成熟度够再启用。
⚠️ 心态校准：目标是"能快速恢复"，不是"第一次就做对"。后者不可能，前者可以练。

Part 3 · Beyond

每天都推送至主线，未完成的功能往哪藏？

01
CI 要求主线随时可部署，但未完成代码也要每天合并入主线——这个悖论靠三种技术解决。

🚩 Feature Toggle

代码里加配置开关，动态启用或禁用功能。

  • 未完成功能默认关
  • 按环境或用户分组开放
  • 上线后可灰度、可紧急关闭

🌿 Branch by Abstraction

大规模重构时的"不开分支的分支"。

  • 引入抽象层，新旧实现并存
  • 通过配置切换，逐步迁移
  • 完成后删旧实现和抽象层

🌓 Dark Launch

新功能部署到生产但对用户不可见。

  • 只做内部监控和压测
  • 观察性能和稳定性
  • 信心足够后再开放
共同点：三招都在做同一件事——把"代码部署"和"功能发布"解耦，让主线既可随时部署，又能容纳未完成的工作。

Part 3 · Beyond

CI → CD → 持续部署 · 自动化边界一次外推一层

02
三个概念常被混用，差别就在"哪一步还需要人手动按按钮"。
1
持续集成 (CI) 自动 构建 + 快速测试  手动 后续测试、部署、发布 — 目标是主线始终处于"已集成、已初步验证"。
2
持续交付 (CD) 自动 完整流水线到类生产环境  手动 只剩"上生产"这一个按钮 — 主线始终处于"可随时发布"。
3
持续部署 (Continuous Deployment) 自动 通过所有阶段就直接上生产  人工 只留紧急刹车 — 发布变成一个无感事件。
不必一路走到底：CI 是 CD 的前提，CD 是持续部署的前提。每多一层自动化，需要更高的测试信心——团队按当前能力选择停在哪一层。

为什么"在 feature branch 上跑 CI"不算 CI

03
CI 的"集成"字面意思是——代码合并入主线。仅是自动运行测试，不等于真正的 CI。
维度 长期特性分支 基于主干的 CI
分支寿命 数天 — 数月 ≤ 1 天
合并时机 功能完成才合并 每天至少一次合并回主线
主线状态 落后于多个开发者的并行工作 始终是团队的当前状态
冲突规模 大而难 小而易
叫 CI 吗？ 只是"分支上的自动化构建" ✅ 是
判定方法：不看有没有使用 Jenkins/GitHub Actions，而看每天是否真正将代码合并回主线。短期 feature branch（< 1 天）与 CI 兼容，长期分支不兼容。

前置 PR 评审与 CI 的张力

04
评审要时间，CI 要快速反馈——这两件事天然打架。

⚠️ 前置评审的代价

  • 评审者响应慢 → 集成延迟数小时甚至数天
  • 打破"每天至少合并一次"的节奏
  • 积累的变更量越大，评审反而越难做
  • 长期看，退化成"特性分支 + CI 工具"

✅ CI 团队的四种应对

  • 结对编程：写的时候就是实时评审
  • 集成后评审：先合并入主线，问题靠后续提交修复
  • 分类处理：重要变更走重评审，琐碎变更走轻流程
  • 紧密小团队重估"是否每次都要重量级 PR"
不是"必须放弃 PR"：是看清"每个 PR 都要前置评审"这个默认值，是不是在悄悄扼杀 CI 的节奏。根据团队具体情况选流程。

Problem · 问题

CI → CD → 持续部署，自动化边界一次外推一层。CI 是进化式设计和安全重构的支柱——与自测试代码互相成全。
