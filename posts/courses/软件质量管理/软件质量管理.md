# 软件质量管理（2.pdf）小测复习（用例子讲清楚版）

这份课件的主线可以理解为：先指出瀑布在现实项目里为什么经常失灵，然后给出一套更贴近不确定性的做法（用户故事/估算/Scrum），再把“质量”落到可交付增量、流动效率（看板/WIP/瓶颈）、评审与测试上，最后延伸到 AI Agent 时代质量与流程约束的变化。

---

## 1) 为什么需要敏捷：瀑布模型到底哪里错了？

瀑布模型最容易被考的点不是“阶段有哪些”，而是它的关键前提：**开始开发之前，需求是清晰、完整、稳定的**。课件要你记住：现实里需求往往做不到。

### 例子：做一个校园二手交易小程序

- 瀑布的做法：先写完完整需求文档，做完设计，再开始开发，最后测试上线。
- 现实发生的事：
  - 第 1 周：同学只说“能发布商品、能聊天”。
  - 第 3 周：学校要求实名、举报、风控；聊天要加敏感词过滤。
  - 第 5 周：发现大家更想要“同城面交点/地图”，需求变了。
- 如果你还按瀑布：你会在“后期”反复返工（设计/代码/测试一起返），成本很高。

你要能用一句话总结：**敏捷想解决的是“需求变化不可避免”这件事，用更短周期的反馈来降低返工损失。**

## 2) 用户故事与验收标准：怎么把“需求”写得可做、可测、可沟通？

### 2.1 用户故事是什么？怎么写？

常用模板：**As a … I want … so that …（作为…我想要…从而…）**

#### 例子：支付后能看到订单

用户故事：

> 作为一名用户，我想在支付成功后看到订单详情，从而确认我买到了什么以及何时发货。

这句话解决的是“谁要什么价值”，但还不够让团队直接开发与测试。

### 2.2 验收标准（Acceptance Criteria）是什么？有什么用？

验收标准是把“故事”变成“可以判定对错的条件”，一般写给开发/测试/PO共同对齐。

#### 例子：给上面的用户故事配验收标准

- 当支付成功后，页面自动跳转到订单详情页
- 订单详情页至少展示：商品名、数量、金额、订单号、支付时间、订单状态
- 支付失败时提示用户重试，并且不会生成“已支付”订单
- 网络超时的情况下，用户能在“我的订单”里查到该笔订单的最终状态

你要会说清楚：**用户故事回答“做什么/为了什么价值”，验收标准回答“怎样才算做对”。**

### 2.3 INVEST：怎么判断一个用户故事写得好不好？

INVEST 是小测高频。建议用同一个故事来解释每个字母。

#### 例子：把“支持微信支付”写成一个好故事

坏写法（太大、不可测）：  
> 支持微信支付。

更好的用户故事：  
> 作为一名用户，我想使用微信支付完成购买，从而快速结算。

用它解释 INVEST：

- Independent（独立）：不要依赖一堆前置故事才能开始做（例如“先把登录/下单都做完”才有意义）
- Negotiable（可协商）：不是合同条款，可以谈实现方式（比如先做 H5 支付，再做小程序原生）
- Valuable（有价值）：对用户/业务有价值，不是“为了技术而技术”
- Estimable（可估算）：团队能判断复杂度；如果“支持微信支付”包含退款、分账、风控就很难估
- Small（足够小）：能在一个 Sprint 内完成；如果做不到要拆分（先做支付成功路径，再补失败/超时/退款）
- Testable（可测试）：能写出验收标准与测试用例（成功/失败/重复支付/回调异常等）

### 2.4 用户故事地图（Story Mapping）：为什么要“全景”？

故事地图解决的是：你拆需求拆得很碎，但还能看见“用户完整旅程”。

#### 例子：二手交易的用户旅程

- 旅程主线：浏览商品 → 选择商品 → 下单支付 → 联系卖家 → 交易完成 → 评价/售后
- 每一步下面再列故事：例如“浏览商品”下面有搜索、筛选、推荐、收藏

考试时你不需要画得很精美，但要能说明：**它帮助你在拆分时不丢主线，也方便规划版本优先级。**

## 3) 估算与计划：为什么估算常错，但过程仍然有用？

课件强调：估算不等于预测未来的水晶球，估算的价值在于让团队把不确定性摊开讨论。

### 3.1 工时估算 vs 相对估算（Story Point）

#### 例子：同一个需求，不同人的“工时”会很离谱

需求：做一个“订单列表页”。

- 新手说：2 天（第一次做，不熟）
- 熟手说：半天（做过类似的）

如果你用 Story Point，相对估算会更稳定：  
“订单列表”比“订单详情”简单一档，比“支付接入”简单两档。

### 3.2 T 恤尺码：什么时候用？

#### 例子：快速粗筛 Backlog

把待办条目先按 S/M/L/XL 分桶：

- S：改文案、加一个字段展示
- M：一个小页面 + 接口改动
- L：跨多个模块、需要评审与联调
- XL：看起来像一个子项目，应该先拆

### 3.3 计划扑克（Planning Poker）：流程题常考

#### 例子：团队用计划扑克估算“订单列表”

1. PO 讲清楚故事与验收标准
2. 团队提问，把不确定点问出来（接口？分页？筛选？异常状态？）
3. 每人独立出牌（例如 3/5/8 点）
4. 分歧最大的人解释原因（出 8 的人可能想到“还要做筛选与缓存”）
5. 再次出牌，直到收敛

你要会说清楚：**计划扑克的目的不是算准，而是把隐藏复杂度暴露出来，让团队形成共同理解。**

## 4) Scrum：用一个完整 Sprint 把角色/事件/工件/承诺串起来

### 4.1 用例子理解“三个支柱”

#### 例子：Sprint 中怎么体现透明/检视/适应？

- 透明：Backlog、DoD、当前进度对团队可见（看板、燃尽图都属于手段）
- 检视：Daily Scrum 看进度与阻碍，Review 看增量是否满足目标
- 适应：发现偏离目标就调整计划（拆分故事、换实现、调整范围）

### 4.2 角色（Roles）用例子讲清楚

#### 例子：一个 Sprint 团队

- Product Owner（PO）：决定“这个 Sprint 最重要的价值是什么”（比如先把支付成功路径打通）
- Scrum Master（SM）：保障会议有效、移除障碍（比如推动外部团队给支付回调权限）
- Developers：负责实现增量并满足 DoD（包括开发、测试、集成）

### 4.3 事件（Events）用例子讲清楚

#### 例子：两周 Sprint 的典型节奏

- Sprint Planning：选定 Sprint Goal（如“用户能完成下单支付并在订单详情查看状态”）
- Daily Scrum：每天同步阻碍（比如“支付回调环境还没通”）
- Sprint Review：演示可工作的增量给干系人看（能下单、能支付、能查订单）
- Sprint Retrospective：复盘改进（比如“验收标准写得太模糊导致返工”）

你要能把每个事件说成一句“目的句”，而不是只背名称。

### 4.4 工件（Artifacts）与承诺（Commitments）用例子讲清楚

#### 例子：同一个目标下的工件与承诺

- Product Backlog + Product Goal：产品长期目标（例如“学期内完成校园交易闭环”）
- Sprint Backlog + Sprint Goal：本 Sprint 的目标（例如“支付+订单查询闭环”）
- Increment + Definition of Done（DoD）：实际交付成果 + 它必须满足的质量门槛

## 5) DoD vs 验收标准：两个最容易混的概念，用例子一次分清

### 例子：同一个故事的验收标准与团队 DoD

用户故事：支付成功后跳转订单详情。

验收标准（这条故事“对不对”）：

- 支付成功后自动跳转到订单详情页
- 订单详情展示金额、订单号、状态

DoD（团队对“完成”的统一底线）可能是：

- 代码合并到主分支
- 自动化测试通过（至少覆盖支付成功/失败）
- 关键逻辑有评审通过
- 部署到测试环境并通过验收

你要会用一句话答题：  
DoD 是“团队对交付增量的统一完成门槛”，验收标准是“某条用户故事的可接受条件”；故事满足验收标准但没满足 DoD，仍然不能算真正可交付。

## 6) 看板 / 价值流 / 瓶颈：为什么“在做太多事”会拖垮质量？

这部分经常用来考你对“流动”的理解：不是人都很努力就会快，**队列与等待**会吞噬效率，还会让质量问题更晚暴露。

### 6.1 WIP 限制是什么？为什么能解决问题？

WIP（Work In Progress）就是“在制品数量”，WIP 限制就是给“同时进行的工作数量”设上限。

#### 例子：没有 WIP 的评审队列

假设一个团队每天能评审 5 个 PR，但 Agent/开发每天能产出 10 个 PR：

- 第 1 天：待评审 10 个
- 第 2 天：待评审 15 个
- 第 3 天：待评审 20 个

后果不是“代码更多了”，而是：

- PR 越积越多，评审越来越晚，缺陷发现越来越晚
- 每个 PR 的上下文都更难捡起来，评审质量下降

如果设定 WIP：例如“待评审最多 10 个”，一旦满了就必须先消化评审，再继续产出新 PR，队列就不会无限膨胀。

### 6.2 瓶颈漂移与 TOC：为什么优化非瓶颈没用？

TOC（约束理论）的关键话术：**系统吞吐由瓶颈决定，优化非瓶颈不会提升整体产出。**

#### 例子：编码不再是瓶颈，评审变成瓶颈

- 你把编码从“每天 5 个需求”优化到“每天 10 个需求”（比如引入 Agent）
- 但评审仍然只能处理“每天 5 个需求”

最终上线速度还是每天 5 个需求，剩下的都卡在评审队列里。这就是“瓶颈漂移”：当你把一个环节变快，瓶颈会转移到另一个环节。

## 7) 评审（代码评审/审查）：为什么它既是质量关口，也是知识机制？

### 7.1 评审的两层目标

- 质量层：提前发现缺陷与风险（逻辑 bug、安全问题、可维护性问题）
- 组织层：传递隐性知识，让团队对系统形成共同理解

### 7.2 例子：同一个 PR，评审为什么能“拦住事故”？

场景：支付回调处理 PR。

- 开发/Agent 写了“回调成功就把订单置为已支付”
- 评审者指出：回调可能重复到达，必须幂等；还要校验签名、防重放

如果没有评审，测试可能也不一定覆盖到“重复回调/恶意回调”的边界情况，线上风险很大。

### 7.3 理解力负债（Comprehension Debt）用例子讲清楚

“理解力负债”可以用一句话解释为：系统里存在大量“别人很难快速理解的代码/决策”，导致后续维护、评审、排障越来越慢。

#### 例子：变量命名与结构让人看不懂

- Agent 产出一段能跑的代码，但命名含糊、结构混乱、缺少关键边界处理
- 每次评审都要花更多时间“先理解它在干嘛”，吞吐降低，缺陷更容易漏掉

所以课件强调评审不仅是“过一道门”，还要让代码更容易被团队理解。

## 8) AI Agent 时代：哪些东西会变难？哪些东西反而更重要？

课件后段的核心观点可以总结为：当产出速度变快时，新的约束往往不是“写代码”，而是“定义清楚、验证充分、评审跟得上、透明可追踪”。

### 8.1 为什么 Backlog 粒度要重新校准？

#### 例子：同一个故事，人做 vs Agent 做的节奏不同

- 人做：一个故事可能要 2~3 天，粒度大一点也能接受
- Agent 做：一天内可能产出多个 PR，如果故事太大，会出现：
  - 一次性产出太多改动，评审压力爆炸
  - 你很难确认“哪些已验证，哪些只是生成了代码但没测”

所以更合适的做法是：把条目写得**边界更清晰、验收标准更明确**，让评审可以快速判断对错。

### 8.2 为什么 INVEST 里的 Estimable（可估算）会被冲击？

#### 例子：你以为是 1 天，结果评审与返工拖成 1 周

Agent 很快写出实现，但真实交付需要：

- 补测试（或把测试规格写清楚并落实）
- 修复边界问题（异常、并发、权限、安全）
- 通过评审与上线流程

因此“估算”不再是“写完代码要多久”，更像是“从需求到可交付增量（满足 DoD）要多久”。估算的对象变了，估算方式也要跟着调整。

### 8.3 为什么 Testable/TDD 会从“最佳实践”变成“生存必需品”？

#### 例子：没有测试的 Agent 产出，靠人工点点点会崩溃

Agent 一天产出 10 个 PR，如果每个 PR 都靠人工回归：

- 测试成本爆炸
- 漏测概率上升
- 评审者也不敢放行（风险不可控）

如果你有明确的测试规格 + 自动化测试（TDD/BDD 思路都可以作为方法），评审者能更快相信结果，吞吐会明显提升。

### 8.4 新透明需求：不仅“做了什么”，还要“验证了什么”

#### 例子：一个条目要明确两类状态

- 已验证：哪些测试跑过？覆盖了哪些场景？
- 待评审：哪些 PR 还没过人审？哪些只是生成但未合并？

课件强调：当“待评审”持续膨胀时，团队要做决策（暂停新产出/调配更多人评审/引入辅助评审等），核心是把系统从失控拉回可控。

## 9) 扩展/规模化敏捷：LeSS、SAFe、BDD 用例子记住“一句话”

这部分小测常见形式是“看到缩写，能说出它大概解决什么”。

### 例子：一个产品从 1 个团队扩到 8 个团队

- LeSS：更偏“多个 Scrum 团队如何协同交付同一个产品”，强调尽量保持 Scrum 的简单性
- SAFe：更偏“大型组织规模化敏捷”的框架，强调跨团队/跨层级的对齐与节奏（例如更清晰的规划层次）
- BDD：更偏“需求到测试的表达方式”，用行为场景（Given/When/Then）把验收写得更可测、更可沟通

## 小测自测题（每题给一个“能直接背/能直接写”的示范答案）

### 1) 软件质量、QA、QC 分别是什么？怎么分工？举例

示范答案（写作模板）：

- 软件质量：软件满足用户需求与约束的程度（功能正确、可靠、安全、可维护等维度）
- QA（质量保证）：**面向过程**，确保过程能持续产出高质量结果（制定标准、审核流程、推动改进）
- QC（质量控制）：**面向结果**，对产品进行检验/测试/评审来发现缺陷

例子：做支付功能

- QA：定义 DoD、要求关键路径必须有自动化测试、规定 PR 必须评审、定期检查流程是否被遵守
- QC：写测试用例、执行回归、做代码评审、验证支付成功/失败/超时/重复回调

### 2) PDCA 四步是什么？每步产出/证据是什么？

示范答案（写作模板）：

1. Plan：设目标与方案（产出：改进计划、指标、验收口径）
2. Do：按方案执行（产出：实施记录、变更内容、实验数据）
3. Check：检查效果（产出：对比数据、缺陷率/周期变化、评审结论）
4. Act：固化或调整（产出：新标准/新流程、后续改进行动）

例子：把“评审队列过长”作为改进目标

- Plan：设定“待评审不超过 10”的目标与 WIP 规则
- Do：落地看板与 WIP 限制
- Check：比较实施前后 PR 等待时间
- Act：固化规则或调整阈值/增加评审能力

### 3) defect / fault / failure 区别与因果链，举例

示范答案（写作模板）：

- defect：缺陷（设计/代码/需求中的问题）
- fault：故障（缺陷在特定条件下导致的错误状态/错误行为原因）
- failure：失效（对外表现为系统没有提供期望服务）

例子：支付回调幂等没做

- defect：代码没做幂等校验
- fault：重复回调导致重复扣款逻辑被执行
- failure：用户被扣了两次钱（对外服务失效）

### 4) 评审/走查/检查（inspection）目的是什么？输出物有哪些？

示范答案（写作模板）：

- 目的：在早期发现缺陷、降低返工成本；对齐设计与实现；传播知识与统一风格
- 输出物：缺陷/问题清单、改进建议、评审结论（通过/需修改/拒绝）、必要时的行动项与责任人

例子：代码评审输出物

- 评论与修改建议
- 必须修复项（阻塞合并）
- 可选改进项（后续优化）

### 5) DoD 与验收标准区别？为什么 DoD 是硬门槛？举例

示范答案（写作模板）：

- 验收标准：针对单条故事的“可接受条件”
- DoD：团队对增量交付的“统一完成门槛”
- DoD 是硬门槛的原因：没有 DoD，交付质量不一致，缺陷会在后期集中爆发，导致不可控的返工与风险

例子：故事满足验收标准但未满足 DoD

- 验收标准：支付成功能跳转订单详情（功能是对的）
- 但 DoD 没满足：没测试、没评审、没合并到主干
- 结论：不能算可交付增量，风险不可控
